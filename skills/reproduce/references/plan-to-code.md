# plan-to-code：按计划本地实现代码

目标：按 `plan.md` 在本地实现实验代码，并通过最小测试或远程烟测。helix CLI 不替你写代码，coding agent 负责实现和判断。

## 输入

- `plan.md`；
- `setup.md`（repro 类型）；
- 官方实现、第三方复现、相关论文代码（可选，仅作参考）；
- 当前工作区代码、配置、脚本。

## 输出

- 本地实现代码；
- 最小测试或烟测结果；
- `PROGRESS.md` 中本阶段的进展、阻塞和建议确认状态。

## 执行协议

1. 先读 `plan.md` 和 `setup.md`，确认推荐方案、文件结构、验证标准；
2. 先实现核心算法/模型，再实现数据处理和评测，最后补配置/README；
3. 每实现一个组件，立即跑最小测试；
4. 优先 `uv` 管依赖，其次 conda；环境只落实验隔离层；
5. 需要远程 GPU 才能测试时，用 `exp start` 同步，再用 `exp run --session ...` 跑烟测；
6. 一轮结束后记录状态：改动摘要、启动命令、tmux 会话名、远程路径、开始时间；git 开启时记录 commit。

## 参考代码策略

- 官方实现、第三方复现、相关论文代码都只能作为实现参考；
- 原论文规格优先；
- 引入参考代码片段时要记录来源和许可证风险；
- 不把参考 repo 的工程结构原样搬进实验工作区，除非它就是本次明确选用的官方仓库。

## 阶段出口

当代码已按 `plan.md` 实现，并且最小测试/烟测通过后，在 `PROGRESS.md` 写：

- 当前阶段：`C. plan-to-code`；
- 阶段状态：`建议用户确认`；
- 当前阻塞：失败测试、缺失数据、远程资源不足等，没有就写「暂无」；
- 下一步：等待用户确认 C 阶段，或进入 `run-monitor-analyze`。
